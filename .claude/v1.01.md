# Roadmap: Settings/Branding, Per-Service Pages, New Checks, Maintenance Windows, 2FA

## Context

`monsee` (status.cryvex.xyz) has completed Phases 1-9 per `CLAUDE.md` (backend clean
architecture, worker, frontend admin + public status page). The user wants six new
capabilities, continuing as **Phases 10-15**:

1. **Phase 10** — Admin-configurable Settings: site title, logo URL, and a runtime
   toggle for the public status page (replaces the build-time `NEXT_PUBLIC_STATUS_PAGE`
   / `PUBLIC_STATUS_ENABLED` env vars). "Powered by monsee" footer stays hardcoded.
2. **Phase 11** — Per-service public visibility toggle, per-service uptime-bar toggle,
   a dedicated public page per service (`/s/<slug>`) that's independently toggleable,
   optional custom domain per service (Host-based routing via Next.js middleware),
   and an admin-configurable uptime history range. Uptime history is exposed via API.
3. **Phase 12** — Admin-only "Overview" page: every service with a simple uptime bar
   (reuses Phase 11's component/API), always visible to admins regardless of public
   toggles.
4. **Phase 13** — Three new monitor check types: SSL/TLS certificate expiry, keyword
   match (HTTP body contains/not-contains), DNS resolution — all stdlib-only, no new
   infra.
5. **Phase 14** — Maintenance windows per service: public banner, excluded from
   uptime stats and incident auto-creation.
6. **Phase 15** — Optional per-user TOTP 2FA with backup codes, added as a second
   login step.

Plus five follow-on features the user selected, as **Phases 16-20**:

7. **Phase 16** — Incident updates / timeline (Investigating → Identified → Monitoring
   → Resolved), shown on the public incident view; auto-incidents also seed the
   timeline.
8. **Phase 17** — Embeddable SVG uptime badge per service + incidents RSS feed.
9. **Phase 18** — Manual status override per service (pin status independent of checks).
10. **Phase 19** — Audit log viewer (read UI over the existing `audit_log` table).
11. **Phase 20** — Per-monitor response-time (latency) sparkline.

All decisions below were confirmed with the user across multiple rounds of questions
(custom domains: yes; admin overview: simple per-service uptime graphs, not
overcomplicated; follow-on features: incident timeline + badge/RSS + manual override +
audit viewer + latency charts; explicitly **not** email subscribers or service groups).

Migrations continue from `000009` → `000010`-`000016` (7 new migration pairs; Phases
17, 19, 20 add no tables).

## Cross-Cutting Refinements (read first — these touch multiple phases)

These were uncovered while validating the draft against the actual code; they prevent
real breakage and reduce avoidable DB load.

1. **CSP must allow external images.** `frontend/next.config.ts` currently sets
   `img-src 'self' data:`. The Phase 10 "paste a logo URL" feature serves an arbitrary
   external host, which this CSP **blocks**. Change to
   `img-src 'self' data: https:` (and keep `data:` for any inlined assets). Without
   this, custom logos silently fail to render on the public page, login, and sidebar.

2. **`FrontendURL` is live — do not remove it.** `cmd/worker/main.go:66` passes
   `cfg.FrontendURL` into `notifications.NewDispatcher` for alert back-links. The
   draft's Phase 10 note calling it dead code was wrong. Only
   `cfg.PublicStatusEnabled` / `PUBLIC_STATUS_ENABLED` becomes dead (replaced by the
   DB-backed toggle) and may be removed.

3. **Settings hot-path caching.** Phase 10 introduces a `Settings.Get(ctx)` call on
   the public `/api/v1/status`, `/api/v1/incidents`, and `/api/v1/services/*` paths
   (the "is the public page enabled?" guard) plus `/api/v1/settings` itself. To avoid
   a DB round-trip per public request, give `SettingsService` a tiny in-process cache
   (e.g. `sync.RWMutex` + `cachedAt time.Time`, 30s TTL) invalidated on `Update`. This
   is the only service that needs it because it's read on every public request.

4. **Next.js server fetches must opt out of caching.** All new server-component
   fetches of `/api/v1/settings` / `/api/v1/services/*` (in `app/layout.tsx`,
   `app/page.tsx`, `app/s/[slug]/page.tsx`, `app/s/by-domain/[domain]/page.tsx`) must
   pass `{ cache: "no-store" }` so toggles/branding changes take effect immediately
   (pages are `force-dynamic`, but `fetch` still caches by default in Next 15).

5. **`check_results` retention vs uptime range.** The `TaskCleanupOldResults` task
   constant exists but is **not** registered in the worker mux (`cmd/worker/main.go`
   only wires `TaskRunMonitorCheck`) and **not** scheduled — so results are currently
   never pruned and 90-day uptime data is safe. Guardrail: if a cleanup job is ever
   wired, its retention window must be `>= max(uptime_range_days)` (cap is 365) or the
   uptime bar loses history. Note this in `CLAUDE.md` rather than building cleanup now.

6. **`UpdateMonitor` is a full-replace (positional `$1..$12`), not COALESCE.** The
   frontend always submits the entire monitor form, so adding the Phase 13 columns
   means extending the positional query and ensuring the form always sends them. Do
   **not** convert it to narg (keep the existing convention); just append the new
   columns positionally and map them in `repository/postgres/monitor.go`'s
   `Create`/`Update`.

7. **Avoid Fiber route collisions (param vs static, and two param names at one
   position).** The naïve route names collide:
   - `GET /api/v1/services/by-domain` would be captured by `GET /api/v1/services/:slug`.
   - `GET /admin/services/uptime` (batch) would be captured by
     `GET /admin/services/:id/uptime`'s `:id`.
   - You cannot register two *differently named* params (`:slug` and `:id`) at the
     same path segment position.
   **Resolution — use these final route names** (also reflected in each phase below):
   - Dedicated page by slug → `GET /api/v1/pages/:slug` (separate prefix).
   - Custom-domain lookup → `GET /api/v1/by-domain?domain=<host>` (top-level, no
     `:slug` tree).
   - Public per-service uptime → `GET /api/v1/services/:id/uptime` (by **id**).
   - Admin single uptime → `GET /admin/services/:id/uptime`; admin batch → `GET
     /admin/uptime` (separate path, not under `/services`).
   The aggregate `/` page fetches uptime by **id** (the `/api/v1/status` payload
   already carries `id`); the dedicated page resolves slug→service via `/pages/:slug`,
   then fetches uptime by the returned `id`.

8. **Slug/custom_domain lifecycle.** The unique indexes are partial
   (`WHERE ... IS NOT NULL`), so an **archived** service still holds its slug/domain
   and blocks reuse. In `ArchiveService` (`db/queries/services.sql` /
   `repository/postgres/service.go`), also `SET slug = NULL, custom_domain = NULL` so a
   new service can take them. Auto-derive a default slug from the service name on
   create (frontend slugify: lowercase, spaces→`-`, strip non `[a-z0-9-]`), editable
   before save — saves the admin from hand-typing.

9. **Custom-domain host must not expose admin.** The Phase 11 middleware rewrites only
   `/` on a non-app host. Harden it: on a non-app host, any non-status path
   (`/admin/*`, `/login`, `/api/*` except the public status calls the page makes)
   should `redirect` to the primary `APP_URL` host (or 404), so a customer's vanity
   domain only ever serves that one service's status page.

10. **Settings cache is per-process.** The 30s TTL cache (refinement #3) is not shared
    across backend replicas; an `Update` on one instance is invisible to others until
    their TTL lapses. Fine for the current single-backend compose; if scaled out, drop
    the cache or move to a Redis-backed cache. Document, don't over-engineer now.

## Phase 11 — Per-Service Visibility, Dedicated Pages, Custom Domains, Uptime History (#2)

### Migration `db/migrations/000011_service_visibility.up.sql` / `.down.sql`
```sql
ALTER TABLE services
  ADD COLUMN public_visible         BOOL NOT NULL DEFAULT true,
  ADD COLUMN show_uptime            BOOL NOT NULL DEFAULT true,
  ADD COLUMN dedicated_page_enabled BOOL NOT NULL DEFAULT false,
  ADD COLUMN slug                   TEXT,
  ADD COLUMN custom_domain          TEXT,
  ADD COLUMN uptime_range_days      INT NOT NULL DEFAULT 90;

CREATE UNIQUE INDEX services_slug_unique ON services (slug) WHERE slug IS NOT NULL;
CREATE UNIQUE INDEX services_custom_domain_unique ON services (custom_domain) WHERE custom_domain IS NOT NULL;
```
- `public_visible` — shown on the aggregate `/` status page.
- `show_uptime` — uptime bar shown (on `/` and the dedicated page), independent toggle.
- `dedicated_page_enabled` + `slug` — enables `/s/<slug>`.
- `custom_domain` — optional; presence is its own enable flag. The by-domain route
  serves the service whenever `custom_domain` matches the request Host, independent of
  `dedicated_page_enabled`/`slug` (so a customer can use only a vanity domain without a
  public slug). It still respects `public_visible` for whether the service renders.
- `uptime_range_days` — admin-configurable (e.g. 7/14/30/60/90), default 90.

### Backend — uptime history
- `db/queries/check_results.sql` — add:
  ```sql
  -- name: GetDailyUptimeForMonitor :many
  SELECT
    date_trunc('day', checked_at)::date AS day,
    count(*)                                    AS total,
    count(*) FILTER (WHERE status = 'up')       AS up_count,
    count(*) FILTER (WHERE status = 'down')     AS down_count,
    count(*) FILTER (WHERE status = 'degraded') AS degraded_count
  FROM check_results
  WHERE monitor_id = $1
    AND checked_at >= now() - ($2 || ' days')::interval
  GROUP BY day
  ORDER BY day ASC;
  ```
- `internal/domain/monitor.go` — add `DailyUptime{Day time.Time; Total, Up, Down,
  Degraded int32}` and `ListDailyUptime(ctx, monitorID string, days int32)
  ([]*DailyUptime, error)` to `CheckResultRepository`.
- `internal/repository/postgres/check_result.go` — implement `ListDailyUptime`.
- New `internal/service/uptime.go` — `UptimeService{monitors, checkResults}`,
  `GetServiceUptime(ctx, serviceID string, days *int32) (*domain.ServiceUptime, error)`:
  loads the service (for default `uptime_range_days` if `days` is nil) and its
  monitors, calls `ListDailyUptime` per monitor, and for each day derives a display
  `status`: `down` if `down_count>0`, else `degraded` if `degraded_count>0`, else `up`
  if `total>0`, else `no_data`; `uptime_percent = (up+degraded)/total*100`.
- Wire `UptimeService` into `cmd/server/main.go` and `handler.Deps` (`Uptime
  *service.UptimeService`).

  import { NextRequest, NextResponse } from "next/server";

  export function middleware(req: NextRequest) {

    const appHost = new URL(process.env.APP_URL ?? "http://localhost:3000").host;
    if (req.nextUrl.pathname === "/" && host !== appHost) {
      const url = req.nextUrl.clone();
      url.pathname = `/s/by-domain/${encodeURIComponent(host)}`;
      return NextResponse.rewrite(url);

    return NextResponse.next();

### Backend

- `db/queries/monitors.sql` — extend `CreateMonitor` (currently positional `$1..$12`)
  and `UpdateMonitor` (currently full-replace `$1..$12`, no COALESCE) with the 5 new

- `internal/checks/runner.go` — add `case "ssl"`, `case "keyword"`, `case "dns"`

- New `internal/checks/ssl.go` — `CheckSSL(ctx, SSLCheckParams{Host string; Port int;

  dialer/timeout, read `conn.ConnectionState().PeerCertificates[0].NotAfter`; `"down"`
  if already expired, `"degraded"` if `daysUntil <= ExpiryThresholdDays`, else `"up"`.
  `ResponseTimeMs` = dial duration. Default port 443 if `m.Port` is nil.
- New `internal/checks/keyword.go` — `CheckKeyword(ctx, KeywordCheckParams{URL,
  Method string; TimeoutMs int; ExpectedStatus int; Keyword string; ShouldExist bool;
  DegradedThresholdMs int})`: like `CheckHTTP` but reads the body (`io.ReadAll` capped
  at e.g. 1 MiB via `io.LimitReader`), `strings.Contains(body, Keyword)`; `"down"` if
  presence doesn't match `ShouldExist` (with a descriptive `Error`), else same
  status/degraded logic as `CheckHTTP`.
- New `internal/checks/dns.go` — `CheckDNS(ctx, DNSCheckParams{Host, RecordType
  string; ExpectedValue *string; TimeoutMs int})`: `net.Resolver` with
  `context.WithTimeout`; dispatch `LookupHost`/`LookupCNAME`/`LookupMX`/`LookupTXT`/
  `LookupNS` by `RecordType`; `"down"` on lookup error or empty result, or if
  `ExpectedValue` is set and not found among results; else `"up"`. `ResponseTimeMs` =
  lookup duration.

### Frontend
- `types/index.ts` — `MonitorType = "http" | "tcp" | "ssl" | "keyword" | "dns"`; add
  the 5 optional fields to `Monitor`.
- `app/admin/services/[id]/page.tsx` — extend `defaultForm`, `openEdit`,
  `handleSubmit`, and the `Select` for `type` (add SSL/Keyword/DNS options), plus
  conditional field blocks (same `form.type === "..."` pattern already used for

  - **ssl**: Host, Port (default 443), "SSL Expiry Threshold (days)".

  - **dns**: Host (domain to resolve), "Record Type" `Select` (A/AAAA/CNAME/MX/TXT/NS),
    "Expected Value (optional)".

## Phase 15 — Two-Factor Authentication (#5)

### New dependency
- `github.com/pquerna/otp` (add to `backend/go.mod`) — `otp/totp` for
  `Generate`/`GenerateOpts`/`Validate`. No QR-image generation needed server-side —
  the frontend renders the QR from the returned `otpauth://` URI.

### Migration `db/migrations/000014_user_totp.up.sql` / `.down.sql`
```sql
ALTER TABLE users
  ADD COLUMN totp_secret       TEXT,             -- AES-256-GCM encrypted base32 secret
  ADD COLUMN totp_enabled      BOOL NOT NULL DEFAULT false,
  ADD COLUMN totp_backup_codes TEXT[];           -- SHA-256 hashes, one-time use
```

### Backend
- `internal/domain/user.go` — add `TOTPEnabled bool` to `User` (JSON-exposed); keep
  `totp_secret`/`totp_backup_codes` out of the JSON struct entirely (load via
  dedicated repo methods, not the main `User` struct).
- `db/queries/users.sql` — add `SetTOTPSecret`, `EnableTOTP` (`SET totp_enabled=true,
  totp_backup_codes=$2`), `DisableTOTP` (`SET totp_enabled=false, totp_secret=NULL,
  totp_backup_codes=NULL`), `GetTOTPByUserID :one` (`totp_secret, totp_enabled,
  totp_backup_codes`), `RemoveBackupCode` (array element removal via
  `array_remove`).
- `internal/repository/postgres/user.go` — implement the above on `UserRepo`,
  extending `domain.UserRepository`.
- New `internal/service/twofactor.go` — `TwoFactorService{users domain.UserRepository,
  encKey []byte}`:
  - `InitiateSetup(ctx, userID) (secret, otpauthURI string, err error)` — `totp.Generate`
    with `Issuer: "monsee"` (or Phase 10 `site_title`) + user email; encrypt secret
    with `pkg/encrypt.Encrypt(encKey, secret)`; store via `SetTOTPSecret` (not yet
    enabled).
  - `ConfirmSetup(ctx, userID, code string) (backupCodes []string, err error)` —
    decrypt stored secret, `totp.Validate(code, secret)`; on success generate 10
    random backup codes, hash each with `pkg/hash` (SHA-256, same helper used for API
    keys), `EnableTOTP`, return plaintext codes (shown once — same UX as API keys).
  - `Disable(ctx, userID, password string) error` — verify password via
    `pkg/hash` bcrypt compare, then `DisableTOTP`.
  - `Verify(ctx, userID, code string) (bool, error)` — try `totp.Validate`; on failure,
    hash `code` and check against `totp_backup_codes`, consuming the match via
    `RemoveBackupCode`.
- **Login flow** (`internal/service/user.go` + `internal/handler/user.go`):
  - `Login` (existing): after password check, if `user.TOTPEnabled`, do **not** issue
    the normal session cookie. Instead issue a short-lived (5 min) JWT with a
    `pending_2fa: true` claim (reuse `middleware.Claims` + a new constant TTL) in a
    separate `pending_2fa` cookie, and respond `{requires_2fa: true}`. If
    `!TOTPEnabled`, behavior is unchanged.
  - New `internal/middleware/auth.go` — `RequirePending2FA(secret string)
    fiber.Handler`, mirrors `RequireAuth` but reads the `pending_2fa` cookie and
    checks the `pending_2fa` claim.
  - New `internal/handler/twofactor.go`:
    - `POST /admin/2fa/setup` (RequireAuth, any role) → `InitiateSetup` → `{secret,
      otpauth_uri}`.
    - `POST /admin/2fa/confirm` (RequireAuth) → body `{code}` → `ConfirmSetup` →
      `{backup_codes}`.
    - `POST /admin/2fa/disable` (RequireAuth) → body `{password}` → `Disable`.
    - `POST /auth/2fa/verify` (RequirePending2FA, **not** RequireAuth) → body
      `{code}` → `Verify`; on success, `IssueToken` + set the real `session` cookie,
      clear `pending_2fa` cookie.
  - Router: register `/auth/2fa/verify` in the existing `authGroup` (alongside
    `/auth/login`/`/auth/logout`); register `/admin/2fa/*` in the `admin` group
    (no `RequireAdmin` — per-user feature).
- **Brute-force protection.** A 6-digit TOTP is 1M combos; the global limiter
  (100/min/IP, `redisrepo.NewRateLimiter`) is too loose for `/auth/2fa/verify`. The
  30s TOTP step window limits exposure, but backup codes are long-lived until used.
  Add a stricter per-IP+per-pending-user limit (e.g. 5 attempts / 5 min) on the verify
  endpoint specifically — reuse the existing Redis sliding-window limiter with a
  distinct key prefix (`rl:2fa:<ip>`), or fail the pending session after N bad codes.
- **Secret/backup-code hygiene.** `totp_secret` is AES-256-GCM encrypted at rest
  (reuse `pkg/encrypt` with the same `ENCRYPTION_KEY`). Backup codes are high-entropy,
  so unsalted SHA-256 (the existing `pkg/hash` API-key helper) is acceptable; never log
  or return them except the one-time reveal at `ConfirmSetup`.

### Frontend
- `types/index.ts` — `User.totp_enabled: boolean`; `LoginResponse` (or similar) gets
  an optional `requires_2fa: boolean`.
- `lib/api/auth.ts` — `login()` returns the `requires_2fa` flag; add `verify2FA(code)`,
  `setup2FA()`, `confirm2FA(code)`, `disable2FA(password)`.
- `app/login/page.tsx` — on `login()` response with `requires_2fa: true`, show a
  second step: 6-digit code input → `verify2FA(code)` → on success, proceed to
  `/admin/services` as before.
- New admin page `app/admin/security/page.tsx` (sidebar entry `{ href:
  "/admin/security", label: "Security", icon: ShieldCheck }`, visible to all roles —
  not admin-only): "Two-Factor Authentication" card —
  - If `!totp_enabled`: "Enable 2FA" → `setup2FA()` → render QR (via `qrcode.react`,
    from `otpauth_uri`) + manual secret + 6-digit confirm input → `confirm2FA(code)`
    → show backup codes (copy/download), same one-time-reveal UX as
    `app/admin/api-keys/page.tsx`'s created-key dialog.
  - If `totp_enabled`: "Disable 2FA" → password prompt → `disable2FA(password)`.
